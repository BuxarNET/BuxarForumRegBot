4. **Читаемость** — документация и понятный код

# Протокол Рефакторинга (КРИТИЧНО)
Перед изменением существующего кода:
1. Проанализируй функцию: назначение, вход/выход, побочные эффекты
   1.1 [ ] **Изучи лог:** (если имеется) Проанализируй приложенный лог ошибок, определи затронутые модули и файлы.
   1.2 [ ] **Изучи мои комментарии:** Проанализируй ошибки которые я вижу и какой я хочу получить результат.
   1.3 [ ] **Изучи загруженные файлы с кодом в рамках октрытой сессии** Если считаешь что файл устарел, запроси новый контекст.
   1.4 [ ] **Запроси контекст через команды:** — **не проси файл или текст, а генерируй готовую команду для терминала**, которую я скопирую и выполню.
    - **Формат команд:** Только команды чтения (`grep`, `sed -n 'p'`, `cat`, `head`, `tail`).
    - **Запрещено:** Команды записи/изменения файлов (`sed -i`, `echo >`, `rm`, `mv`).
    - **Пример запроса:**
      ```bash
      # Покажи строки 83-125 для анализа логики
      sed -n '83,125p' src/controllers/registration_controller.py

      # Найди определение функции и 10 строк после
      grep -A 10 "def analyze_current_page" src/controllers/registration_controller.py
      ```
   1.5 [ ] **Жди вывода:** После того как я вставлю результат команды в чат, продолжай анализ.
   1.6 [ ] **Карта зависимостей:** Если видишь вызов внешних методов, запроси их реализацию через команду `grep`, чтобы не сломать зависимости.
2. Оцени риски: может ли изменение сломать работающую логику
3. **ЗАПРОСИ разрешение у пользователя:**
   - Составь текстовую схему текущего функционала
   - Опиши, что функция делает сейчас
   - Объясни, зачем ты хочешь её изменить
   - Опиши потенциальные риски
   - Жди подтверждения перед применением изменений
